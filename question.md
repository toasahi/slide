# PMO想定質問

回答は未記載です。1〜10章の説明をした際に、DS課のPMOから確認されそうな質問を整理しています。

## 1. 背景・目的

- Q.運開分離の一環として、なぜ今アラート対応の見直しを優先する必要があるのか。
- A.運開分離により、障害発生時の一次切り分けを運用チームが担う場面が増えるためです。対象システムが120以上あり、利用する調査手段や確認観点も分散している中で、限られた時間で影響範囲を見極め、復旧に向けた判断を行う必要があります。現状のままでは、対応に必要な知識や確認先が担当者依存となり、調査や切り分けの品質、初動の速さにばらつきが生じやすくなります。そのため、運開分離を進めるこのタイミングで、アラート対応の進め方や記録の残し方を見直し、一定の品質で対応できる運用を整える必要があると考えています。
- 現状の「調査や切り分けの品質、初動の速さにばらつきがある」と言える根拠は何か。
- 何をもって「担当者によらず一定の品質で対応できる状態」と判断するのか。
- PF課として、今回の取り組みで最終的に何を実現したいのか。

## 2. 現状の課題

- 過去の対応事例がまとまっていないことで、実務上どのような支障が出ているのか。
- 記録先が分散していることで、誰がどの場面で困っているのか。
- GitHub Issue も既にある中で、なぜ課題が解消されていないのか。
- 対象システムが120以上あることが、今回の提案にどう関係しているのか。

## 3. 解決策

- 「アラート対応部屋」とは、既存のチャットルームや個別連絡と何が違うのか。
- なぜアラート対応部屋を導入すると、担当者依存が減ると考えているのか。
- 一次対応者が場を立ち上げるとのことだが、開始条件と終了条件は何か。
- 誰がファシリテートし、誰が記録の主担当になるのか。

## 4. 今のやり方との違い

- 現在のやり方を改善するだけではなく、運用の枠組みとして見直す必要があるのはなぜか。
- 記録先を集約すると、Confluence や Jira など既存の記録との関係はどう整理するのか。
- 個別連絡を減らすことで、重要な関係者への連絡漏れは起きないのか。

## 5. 期待される効果

- 期待している効果は、どの指標で測る想定なのか。
- 初動の速さや見落とし削減について、どの程度の改善を見込んでいるのか。
- ナレッジが蓄積されるだけでなく、実際に再利用される運用にできるのか。
- 上位者への説明やエスカレーションがしやすくなることを、どう価値として捉えているのか。

## 6. なぜDS課の参加・知見が必要か

- PF課だけでは判断しきれない具体的な場面は何か。
- DS課に参加してもらうことで、どの判断や対応が早くなるのか。
- DS課が参加できない場合は、どのように対応するのか。
- DS課に求めるのは、情報提供なのか、判断なのか、作業なのか。
- PF課とDS課の責任分界はどう整理するのか。

## 7. DS課にとってのメリット

- ここで挙げているメリットは、PF課だけでなくDS課にも実際に返るものなのか。
- 異動者や新人でも使えるレベルで、記録の質を揃えられるのか。
- 見落としや手戻りが減ることを、どのように確認するのか。
- ポストモーテムで使用する事実報告書の整理に活用できるとあるが、どのような位置づけで使う想定なのか。

## 8. DS課にお願いしたいこと

- PF課の一次切り分けの完了条件は何か。
- 「対応方針が決まらない場合」とは、どの時点を指すのか。
- 招待する対象システムの担当者は、誰がどの基準で判断するのか。
- DS課の担当者に、どの程度の時間的拘束を想定しているのか。
- 調査履歴の記録は、PF課とDS課のどちらが主に担うのか。
- 深夜や休日のアラートでも、同じ運用を求めるのか。

## 9. 負担を抑えた試行範囲

- なぜ最初の対象を Kubernetes のアラートにしたのか。
- 月あたり15件のうち、実際にDS課の参加が必要になるのは何件程度と見込んでいるのか。
- PF課から必要に応じて声がけする運用で、招待判断が属人的にならないのか。
- 試行の継続、拡大、中止を判断する基準は何か。

## 10. アラート対応部屋の手段と今後の展望

- Q.なぜ GitHub Issue を使うのか。Jira や Confluence ではだめなのか。
- A.集約したいのは、すべての情報を一つのツールに寄せることではなく、アラート対応中の記録先を標準化することです。アラート対応部屋（GitHubIssue）は、初動から一時対応までの仮説、確認内容、判断、対応経緯を時系列で残す場として位置づけます。Confluence は、対応後に整理された事実や事実報告書を残す場として使い分けます。Jiraは、恒久対応や改善施策が必要になった場合のタスク管理に利用する、という整理を想定しています。

- GitHub Issue を標準記録先にすると、既存の記録との二重管理は発生しないのか。
- GitHub Issue 上で、テンプレート、ラベル、クローズ条件はどう定めるのか。
- チーム全体へ展開する場合、どの順序で広げていく想定なのか。
- AI による分析では、何をどこまで支援または自動化したいのか。
- AI を活用する場合、情報管理や誤判定のリスクをどう考えるのか。

## 横断的に確認されそうな質問

- 今回の説明で、DS課として何を合意または了承すればよいのか。
- PF課側で事前に整備しておくべき運用ルールやテンプレートは何か。
- この試行がうまくいかなかった場合、どのように見直すのか。
- PF課側で先に改善すべきことと、DS課に協力をお願いしたいことの線引きは何か。
